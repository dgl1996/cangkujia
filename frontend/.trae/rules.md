# 仓酷家开发军规（Trae必须遵守）


## 第1条：方案设计规范

### 1. 唯一性思维（核心战略）
- 必须首先理解项目核心目标（仓酷家：3分钟出图的仓库版草图大师）
- 任何解决方案必须服从唯一性原则：解决当前问题且不影响核心目标
- 禁止为解决局部技术细节陷入无限循环，时刻问"这对3分钟出图目标有帮助吗？"

### 2. 影响范围检查（必须执行）
- 修改前必须回答：会影响哪些其他功能？（列出具体文件名）
- 必须区分：直接影响 / 潜在影响（最多3个）/ 无影响区域
- 禁止只说"修复了XX"而不说"影响了YY"

### 3. 最小改动原则（过度设计禁令）
- 能用配置解决，不动代码；能改一行，不改十行
- 能用简单方案解决的，禁止引入复杂方案
- 统一处理逻辑：同类问题必须用统一函数处理，禁止分散修复。能用统一工具函数处理的，必须用统一的工具函数，避免同一类功能使用不同的方法和代码。

### 4. 精确执行和服从
- 任何执行动作，不管有几个方案，都必须得到明确确认才可以执行。
- 按需求执行，不擅自扩展，不自以为是，任何未经汇报确认的行为都严厉禁止！避免画蛇添足！

### 5. 风险评估与回退（安全保障）
每次修改必须包含：
- **风险评估**：低风险（reload恢复）/ 中风险（需测试多个功能）/ 高风险（系统崩溃）
- **回退方案**：Git撤销命令 或 注释还原方法（具体到行号）
- **预防措施**：至少1个防止复发的方法

## 第2条：调试与验证规范

### 6. 根因分析要求（禁止盲目修复）
- 不再盲目修复：每次必须深入分析根本原因（问5个为什么）
- 不再局部修复：每次必须问"还有类似的地方吗？"
- 建立问题追踪：记录重复出现2次以上的问题，重点分析系统性原因，并在 `docs/bug-log.md` 自动做记录。

### 7. 系统性思考要求
- 建立状态地图：画出完整的选中状态管理图和测试清单
- 从整体上理解系统行为，建立测试清单而不是靠用户发现
- 解决了一个老问题，必须验证没有引入新问题

### 8. 自检验证机制（交付标准）
- 建立功能清单进行结果验证，不光看代码能不能跑通，还要看结果对不对
- 实际测试验证：不只是代码语法检查，必须在浏览器实际测试功能
- 阶段性任务完成时必须包含：成果清单 + 技术方案 + 自检验证结果 + 隐患分析

## 第3条：禁止事项（红线）

- 禁止删除看似"没用"的代码（可能别的功能依赖它）
- 禁止修改全局样式（可能破坏其他页面）
- 禁止修改公共工具函数而不检查谁在用
- 禁止为解决单一细节问题而偏离核心产品目标
- 禁止重复叙述相同内容（造成用户困惑）

## 第4条：输出格式规范（必须遵守）

### 每次提供方案时的标准格式：

#### 1. 改动说明（1句话）
【要改哪里，怎么改，为什么】

#### 2. 影响范围分析（必须回答）
- **直接影响**：当前这个功能本身
- **潜在影响**：可能波及的3个其他功能（列出具体文件名）
- **无影响**：确认不会动的区域

#### 3. 风险评估与回退（必须回答）
- **风险等级**：低/中/高
- **回退方法**：改坏了如何快速恢复（具体到命令或行号）
- **预防措施**：如何防止复发

#### 4. 测试清单（勾选框）
- [ ] 当前功能正常（具体测试步骤）
- [ ] 功能A未受影响（具体验证方法）
- [ ] 功能B未受影响（具体验证方法）

#### 5. 决策点（如有多方案）
- 方案A：[简述]，优点是XX，缺点是YY
- 方案B：[简述]，优点是XX，缺点是YY
- **等待用户决策**：请输入A或B，或提出新方案

## 第5条：阶段性交付报告（任务完成时必须输出）

### 报告结构：
```markdown
## 阶段性开发成果报告：[功能名称]

### 1. 开发成果
- 完成了：[具体功能]
- 技术方案：[使用的技术/方法]

### 2. 自检验证结果
- [x] 测试项1：结果正常
- [x] 测试项2：结果正常
- [ ] 测试项3：存在问题（说明）

### 3. 发现的问题/隐患
- 问题1：[描述] → 建议：[解决方案]
- 隐患1：[描述] → 影响：[未来可能的风险]

### 4. 下一步建议
- 建议1：[下一步做什么]
- 建议2：[需要注意什么]
```

## 第6条：特别警告（Three.js相关）

- 修改材质/几何体前：确认是否共享引用（影响其他对象）    
- 修改scene对象前：确认是否会影响渲染循环    
- 添加事件监听前：确认是否重复添加（内存泄漏）    
- 修改Three.js相关代码后：必须在浏览器实际测试3D渲染效果，不能只看代码不报错

## 第7条：强制工作流程（Superpowers机制）

执行任何开发任务前，必须按以下四阶段执行，未完成当前阶段禁止进入下一阶段。

### 阶段1：分析（必须输出，否则禁止编码）
- [ ] 我理解了核心目标：3分钟出图的仓库版草图大师
- [ ] 我找到了根本原因（修bug时）/ 我明确了功能边界（新功能时）
- [ ] 我确认了影响范围：直接影响[功能A]，潜在影响[功能B、C、D]
- [ ] 我检查了bug-log.md，确认这不是重复问题

### 阶段2：规划（必须输出，否则禁止编码）
- [ ] 最小改动方案：只改[X]个文件，[X]行代码，符合"能改一行不改十行"
- [ ] 风险评估：[低/中/高]
- [ ] 回退方案：改坏了执行[具体命令/操作]即可恢复
- [ ] 预防措施：如何防止复发[一句话]
- [ ] 我确认了：此方案不会偏离"3分钟出图"核心目标

### 阶段3：执行（实际编码）
- [ ] 代码已按军规编写（防御性检查、最小改动、统一逻辑）
- [ ] 未删除任何"看似无用"的代码
- [ ] 未修改全局样式或公共函数（除非确认无影响）

### 阶段4：验证（交付前必须完成）
- [ ] 浏览器实际测试通过（描述测试结果）
- [ ] 功能清单验证完成（所有勾选框已打勾）
- [ ] 我确认了未引入新问题（或新问题已记录）
- [ ] 已更新bug-log.md（如果是bug修复）

### 强制中断规则
如果任何阶段的勾选框无法完成，必须立即停止并报告：
"【中断】无法完成阶段[X]，原因：[具体说明]，需要：[用户决策/额外信息/其他资源]"

## 第8条：知识沉淀机制

### 问题追踪（必须执行）
每次发现（重复出现2次及以上）和解决bug后，自动在 `docs/bug-log.md` 追加记录：

| 时间 | 问题描述 | 根本原因 | 解决方案 | 预防措施 | 重复次数 |
|------|---------|---------|---------|---------|---------|
| 2026-03-XX | [简述] | [一句话] | [一句话] | [一句话] | [数字] |

### 技术决策记录（必须执行）
每次重大技术决策（如引入新库、改变架构），在 `docs/decisions.md` 记录：
- 决策内容：[做了什么选择]
- 决策理由：[为什么选择A而非B]
- 否决方案：[B方案为什么不行]
- 日期：[时间]
